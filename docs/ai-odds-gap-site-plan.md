## 1. 新サイトの目的

**「予想紙評価 × 自社AI評価 × 直前オッズ」の3者のズレ（ギャップ）を抽出し、可視化するAI分析ツール。**

- 競馬の妙味は「評価」と「人気（オッズ）」が一致しない馬に潜む、という前提に立つ。
- 3つの独立した評価軸を突き合わせ、**「評価は高いのにオッズが甘い馬」「評価は低いのに過剰人気の馬」** を機械的に抽出する。
- AK/KI が「予想を見せる」サイトであるのに対し、本サイトは **「評価とオッズの差分そのものを商品にする」** 別ジャンルのツールとする。

### このサイトが答える問い
- どの馬が「評価のわりにオッズが付きすぎ（妙味あり）」か？
- どの馬が「評価が低いのに人気を集めすぎ（危険な人気馬）」か？
- 予想紙とAIの評価が **食い違っている**レースはどれか？（荒れる可能性のサイン）

### このサイトが答えない問い（重要）
- 「どの馬券をいくら買うべきか」は **扱わない**（[買い目を提示しない](../CLAUDE.md) 方針に準拠）。
- あくまで「評価とオッズのズレの提示」までで止め、購入推奨・投資助言はしない。

## 2. AK/KI との役割分担

| サイト | 役割 | 主な出力 | 本サイトとの違い |
|---|---|---|---|
| **AK（analytics-keiba）** | 買い目判断向けの要約評価 | 3項目要約（安定性 / 能力上位性 / 展開利）、印・買い目 | 「予想」を提示する |
| **KI（keiba-intelligence）** | AI詳細分析 | featureScores 6項目の詳細表示 | 「評価の中身」を見せる |
| **本サイト（AI Odds Gap）** | 評価とオッズの**差分**抽出 | ギャップ・ランキング、過剰／過小人気フラグ | 「評価とオッズのズレ」を見せる |

### 棲み分け原則（[cross-project-safety-rules.md](cross-project-safety-rules.md) 準拠）
- **本サイトは shared データの consumer（読み取り専用）**。AK/KI と同列の「表示側」プロジェクトとして扱う。
- 本サイトのために **AK/KI の仕様・shared JSON 構造を変えない**。足りないデータがあれば、その不足を本ドキュメントに記録し、別タスクで shared 側に正本として追加する（表示側で勝手に補完しない）。
- AK/KI と **UI を統一しない**（UI 差は正常。AK/KI 間ですら統一しない方針）。
- 「評価値」の正本は引き続き shared featureScores。本サイトはそれを **再解釈するだけ**で、新しい評価正本を勝手に作らない。

## 3. 使用データ

すべて **既存の `keiba-data-shared` から読み取るだけ**。新規保存・新規契約の追加はこの設計段階では行わない。

| データ | ソース（既存） | 役割 | 備考 |
|---|---|---|---|
| 予想紙評価 | `{cat}/predictions/computer/...`（コンピ指数＋印） | 予想紙評価の正本 | `customScore = 印1×4 + 印2×3 + 印3×2 + 印4×1` を再利用 |
| 自社AI評価 | `{cat}/featureScores/...`（6項目 0〜100） | AI評価の正本 | KI が詳細表示している同じ素材 |
| 補完情報 | `{cat}/racebook/...` | 馬名・騎手・枠などの表示用 | race-data-importer 由来 |
| 結果（検証用） | `{cat}/results/...` | ギャップ抽出の精度検証 | 後追いの的中率評価にのみ使用 |
| **直前オッズ** | **未確定（要調査）** | オッズ正本 | ↓ 下記「直前オッズの取得」参照 |

### 直前オッズの取得（最大の未解決事項）
- 現状の shared JSON に **「直前オッズ」を継続保存している正本が存在するか未確認**（この設計段階では env/token/外部APIの確認は禁止のため未調査）。
- 候補は以下。**実装フェーズで別タスクとして調査・決定する**：
  1. 既存 racebook / predictions に確定前オッズが含まれているか
  2. 結果JSON に確定オッズ（単勝・人気）があり、それを暫定代替に使えるか
  3. 直前オッズ取得を **shared 側の新正本**として追加する（admin/shared を中心に保存）
- **本サイト側に独自のオッズ取得・独自補完を作らない**。正本は必ず admin/shared 側に置く。

## 4. MVP機能

最小構成。「1レース分のギャップを一覧で見せる」ところまで。

1. **レース選択**：日付 × 競馬場 × レース番号を選ぶ。
2. **ギャップテーブル表示**：1レースの全出走馬について以下を1行で表示。
   - 馬番 / 馬名
   - 予想紙評価（customScore またはランク）
   - 自社AI評価（featureScores 由来の総合スコア）
   - 直前オッズ（または暫定で確定オッズ）
   - **ギャップスコア**（↓ 抽出ロジック参照）
   - フラグ：🔵過小人気（妙味）/ 🔴過剰人気（危険）/ ⚪一致
3. **ソート**：ギャップスコア順に並べ替え。
4. **レースサマリー**：そのレースの「評価とオッズの不一致度」を1指標で表示（荒れ度の目安）。

MVPでは **過去日のデータでの後追い表示**から始め、当日リアルタイム性は次フェーズに送る。

## 5. 抽出ロジック案（たたき台）

> すべて暫定。確定前に検証データで重み・閾値を調整する。

### 5-1. 各評価を 0〜100 に正規化
- **予想紙評価 `P`**：`customScore` をレース内で 0〜100 にスケーリング。
- **自社AI評価 `A`**：featureScores 6項目の加重平均をレース内で 0〜100 にスケーリング。
- **オッズ評価 `O`**：直前単勝オッズから「市場の支持度」を 0〜100 に変換（オッズが低い＝支持が高い＝Oが高い）。
  - 例：`O = 100 × (1/odds) / Σ(1/odds)` をレース内で再スケーリング（控除率は無視した近似）。

### 5-2. ギャップスコア
評価（紙＋AI）と市場（オッズ）の差を取る。

```
evalScore = wP × P + wA × A          # 例: wP=0.5, wA=0.5
gap       = evalScore - O            # 正=評価>市場(妙味) / 負=評価<市場(過剰人気)
```

- `gap` が大きく正 → **過小人気（妙味あり）**：🔵
- `gap` が大きく負 → **過剰人気（危険）**：🔴
- `|gap|` が小さい → 一致：⚪

### 5-3. 紙とAIの不一致（裏サイン）
紙とAIが食い違う馬・レースは別軸で抽出。

```
discord = |P - A|                    # 紙とAIの評価乖離
```
- レース全体の `Σ discord` が大きい＝予想が割れている＝荒れやすいレースのサイン。

### 5-4. 閾値（暫定）
- 妙味フラグ：`gap ≥ +20`
- 危険フラグ：`gap ≤ -20`
- 不一致レース：レース平均 `discord ≥ 25`

## 6. AIの役割

- **正規化と重み付けの調整**：評価軸ごとの分布が異なるため、レース特性に応じて `wP/wA` や正規化方法を学習・調整する余地をAIが担う。
- **ギャップの言語化（注意・コメント生成）**：「この馬は紙・AIともに上位だが単勝が甘い」等の **事実ベースの短いコメント**を生成する。買い目・購入推奨は出さない。
- **荒れ度判定**：`discord` とギャップ分布から、レース全体の波乱度合いを推定する。
- **後追い検証**：results と突き合わせ、ギャップフラグが実際にどれだけ機能したかを集計し、重み・閾値の見直しに使う。

### AIがやらないこと
- 買い目・点数・配分の提示（禁止）。
- 「買え／買うな」の断定的な投資助言。

## 7. 課金方式案（月額 / ポイント）

> 案。確定は別途。決済実装はこの設計段階では扱わない。

### 案A：月額サブスク（推奨たたき台）
| プラン | 内容 | 想定 |
|---|---|---|
| 無料 | 当日1レースのみギャップ表示 / フラグ非表示 | 集客・誘導UI（[無料版方針](../CLAUDE.md) に準拠：誘導UIは残すが中身は非開放） |
| 月額（標準） | 全レースのギャップ＋フラグ＋レースサマリー | メイン課金 |
| 月額（上位） | 後追い検証レポート・AIコメント・荒れ度ランキング | ヘビーユーザー向け |

### 案B：ポイント（都度消費）
- 1レースのギャップ詳細＝Nポイント消費。
- 開催日にまとめ買い（パック）で割引。
- 「特定の重賞だけ見たい」ライト層向け。

### 案C：ハイブリッド
- 月額で「閲覧本数の基礎枠」を付与し、超過分はポイント追加購入。

**MVP は決済なしの閲覧専用で立ち上げ**、利用動向を見てから課金方式を選定するのが安全。

## 8. 実装しないこと・禁止事項

### この設計（docs）段階で禁止
- 既存コードの変更（admin / AK / KI / shared すべて）。
- AK/KI の仕様変更。
- shared JSON 構造の変更・新規フィールド追加。
- token / env の確認・追加。
- 実装 PR の作成（**この PR は docs 追加のみ**）。

### 新サイトの設計方針としての禁止（[cross-project-safety-rules.md](cross-project-safety-rules.md) 準拠）
- 本サイト側に **独自のデータ補完・独自の評価正本**を作らない。正本は admin/shared 中心。
- 表示側の場当たり修正で shared データの欠陥を隠さない（不足は記録して別タスク化）。
- AK/KI と UI を統一しない。片方を直して片方が壊れる修正をしない。
- **買い目・購入推奨・投資助言を出さない**（[買い目を提示しない](../CLAUDE.md) 方針）。
- 直前オッズの取得・保存を新サイト単独で完結させない（shared 正本として整備する）。

## 9. 次フェーズ案

1. **オッズ正本の調査・整備**（最優先）：直前オッズの取得元・保存先を確定し、shared に正本として置く別タスク。
2. **データ契約の確定**：本サイトが読む shared パス・スキーマを `*-contract.md` として明文化（recentHorseHistories 契約と同様の運用）。

4. **後追い検証**：results 突き合わせでフラグの的中傾向を集計し、重み・閾値を調整。
5. **AIコメント生成**：事実ベースの短文コメント（買い目なし）を追加。
6. **当日リアルタイム化**：直前オッズの当日取り込みと自動更新。
7. **課金導入**：利用動向を見て案A/B/C から選定し、決済を実装。
